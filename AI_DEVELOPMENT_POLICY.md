# AI_DEVELOPMENT_POLICY.md

> IdeaNotes 项目 AI 开发与提交流程规范（v1.0）

## 1) 核心原则
1. AI 可负责主要编码，但**未验证改动不得合并**。
2. 小步提交：一个任务一个目标，一个目标一个 commit。
3. 高风险改动（权限、登录、支付、删除、配置）必须先人工批准。
4. 所有改动必须可回滚（保留基线 commit + revert 路径）。

## 2) 角色分工
- 主助手：拆解任务、监督执行、汇总风险与结论
- Subagent：实施改动并执行验证
- 项目负责人：审批高风险改动与最终合并

## 3) 标准执行流程
1. 明确目标与范围（允许改哪些目录）
2. 按最小改动原则实施
3. 运行门禁验证
4. 失败先修复一轮，仍失败则输出阻塞与替代方案
5. 输出结构化交付报告（结论/改动/验证/风险/回滚）

## 4) IdeaNotes 门禁标准（Flutter）
提交前至少执行：
- `flutter analyze`
- `flutter test`
- `flutter build apk --debug`（或当前目标平台等价 build）

满足以下条件才可“可提交”：
- 功能目标达成
- analyze / test / build 全通过
- 无未批准高风险改动
- 提供回滚方案

## 5) 失败处理策略
- analyze 失败：先修类型与静态检查错误
- test 失败：优先修复回归，不允许跳过关键用例
- build 失败：优先修复编译链路，必要时缩小改动范围重试

## 6) 提交与 PR 建议
- 分支：`feat/<topic>` / `fix/<topic>` / `chore/<topic>`
- Commit：`feat:` `fix:` `refactor:` `test:` `chore:`
- PR 必须包含：目标、改动范围、验证结果、风险、回滚

## 7) 统一交付模板
1. 结论：可提交 / 不可提交
2. 改动：文件清单 + 说明
3. 验证：analyze / test / build 结果
4. 风险：高/中/低
5. 回滚：`git revert <commit>` 或回退到基线
6. 下一步：1-3 条
